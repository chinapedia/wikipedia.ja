**冪剰余**（べきじょうよ、[英](../Page/英語.md "wikilink"): Modular exponentiation）とは、[冪乗](https://ja.wikipedia.org/wiki/冪乗 "wikilink")の[剰余](https://ja.wikipedia.org/wiki/剰余 "wikilink")のことである。[数論](../Page/数論.md "wikilink")的に重要な概念であるとともに、[計算機科学](../Page/計算機科学.md "wikilink")、特に[暗号理論](../Page/暗号理論.md "wikilink")の分野での応用が重要である。**冪乗剰余**とも呼ばれる。

正の整数  （底）の整数  乗（[冪指数](https://ja.wikipedia.org/wiki/冪指数 "wikilink")）を正の整数  （[法](https://ja.wikipedia.org/wiki/除法#定義 "wikilink")）で割った余りを、「 を法とした  の -冪剰余」と呼ぶ。つまり、冪剰余を求めるとは、次の  を計算することに他ならない。

  -
    \(c := b^e \bmod m\)

例えば、 5, *e* {{=}} 3、*m* {{=}} 13}} の場合、 は  を  で割った余りなので、冪剰余は  となる。

冪指数  2, 3}} に対する -冪剰余は、通常それぞれ[平方剰余](https://ja.wikipedia.org/wiki/平方剰余 "wikilink")、[立方剰余](https://ja.wikipedia.org/wiki/立方剰余 "wikilink")と呼ばれる。

冪剰余は、指数  が負の場合も定義できる。その場合、 を法として  の[逆数](../Page/逆数.md "wikilink")（[モジュラ逆数](https://ja.wikipedia.org/wiki/モジュラ逆数 "wikilink")）となる *d* によって、

  -
    \(b^{e} \bmod m := d^{|e|} \bmod m\)

と定義する。

冪剰余  を求める計算は、たとえ巨大な数であっても難しくはない。一方、 が与えられたとき、指数  を求めることは難しい。このような[一方向性関数](../Page/一方向性関数.md "wikilink")的性質から、冪剰余の暗号での利用についての研究が進んでいる。

## 定義

整数  と  について、 を法とした  の -冪剰余とは、剰余群 における剰余類 　の -冪、つまり

\[[b]^e \in \mathbb Z / m \mathbb Z\] である。しばしば、剰余類の代わりに[代表元](https://ja.wikipedia.org/wiki/同値関係#同値類 "wikilink")  をとって、整数として扱う。そのようなとき、特に  であるような  を代表元として選ぶことが多い。

また、変数  に関する  を法とする合同式

  -
    \(c = x^e \bmod m\)

が解をもつような  を総称して法  に対する -**冪剰余** (residue of e-th power modulo m) と呼び、解をもたないような  は総じて法  に関して -**冪非剰余** (non-residue of e-th power modulo m) であるという。

## 計算方法

### 直截的手法

冪剰余を求める最も直截的な手法は、 を直接計算し、結果を  で割って余りを求める方法である。 [冪乗\#効率的な演算法](https://ja.wikipedia.org/wiki/冪乗#効率的な演算法 "wikilink")にあるように、冪乗の演算には、 回の乗算が必要であり、それに加えて1回の剰余演算を施すことによって冪剰余を求めることができる。

例として、 4, *e* {{=}} 13, *m* {{=}} 497}} の場合、 を計算することを考える。

  -
    \(c := 4^{13} \bmod{497}\)

は  であり、これを  で割ると、剰余の  は  であることがわかる。

強い暗号では、 は最低でも256桁の2進数（10進数では77桁）である。典型的な例として  5 × 10<sup>76</sup>}} と  17}} の場合を考えてみる。この場合、 は77桁であり、 は2桁だが、 は（10進で）1304桁にもなる。コンピュータにそのような計算をさせることはもちろん可能だが、性能は期待できない。 や  の桁数が増えるほど、 は扱いにくくなる。

### 途中で剰余をとる

次の計算方法は、手順を増やすことになるが、結果としてはこのアルゴリズムのほうが能率が良い。

2つの整数  と  があるとき、

  -
    \((a \cdot b) \bmod{m} = ((a\ \mbox{mod}\ m) \cdot (b\ \mbox{mod}\ m)) \bmod{m}\)

である。よって、最後に剰余演算を1回行うのではなく、乗算を行うたびに  を法とした剰余をとっても計算結果は変わらない。

これによって、剰余算の回数が1回から  回に増えるが、乗算および剰余算の計算コストは被演算数の桁数によるので、結果としてはこのアルゴリズムのほうが能率が良い。また一般に、 がその計算機環境における1語の半分以下に収まる大きさの値の場合  の結果が必ず1語に収まるので、[Bignum](https://ja.wikipedia.org/wiki/多倍長整数 "wikilink") を一切使わない能率の良い計算が可能である。

次の例は、基本としてこの手法を利用し、さらに指数法則の  (b<sup>2</sup>)<sup>x</sup>}} を活用したコードであり、ほぼ [C\#](../Page/C_Sharp.md "wikilink") または [C++](../Page/C++.md "wikilink") のコードとなっている。クラス Bignum は任意の巨大な整数を表現できるものとする。引数 b は底、e は指数、m は法である。

``` cpp
Bignum modpow(Bignum b, Bignum e, Bignum m) {

    Bignum result = 1;

    while (e > 0) {
       if ((e & 1) == 1) result = (result * b) % m;
       e >>= 1;
       b = (b * b) % m;
    }

    return result;
}
```

このコードは、にあるものに基づいており、一つの while ループで冪剰余の計算に必要な全ての作業を行っている。

例として、 4, e {{=}} 13, m {{=}} 497}} をこの手法で計算する。e を2進数で表記すると 1101 になる。e が2進数では4桁なので、ループは4回しか回らない。

  - ループに最初に入ったとき、各変数の値は  4, e {{=}} 1101}}（2進数）,  1}} である。e の右端のビットは 1 なので、result は  つまり**4** となる。e は右にシフトされて 110（2進数）となり、b は  すなわち 16 になる。
  - 繰返しの2回目では、e の右端ビットは 0 なので result は **4** のままとなる。e は右にシフトされて 11（2進数）となり、b は  つまり 256 となる。
  - 繰返しの3回目では、e の右端ビットは 1 なので result は  すなわち **30** となる。e は右にシフトされて 1（2進数）となり、b は  つまり 429 となる。
  - 繰返しの4回目では、e の右端ビットは 1 なので result は  すなわち **445** となる。e は右にシフトされて 0 となり、b は  % 497 つまり 151 となる。

e が 0 になるとループは終了し、結果として **445** が得られる。これは前述のアルゴリズムとも一致している。

### さらなる最適化

[Python](../Page/Python.md "wikilink")で右向き[バイナリ法](https://ja.wikipedia.org/wiki/冪乗#効率的な演算法 "wikilink")（指数の左端ビットから順に見ていく方法）を使った例である。

``` python
import math

def bits(integer): #Gets number of bits in integer
   result = 0
   while integer:
      integer >>= 1
      result += 1
   return result
def power(base, exponent, modulo=None): #Allows fast exponentation with and without moduli
   result = 1
   if not modulo:
      iteration = bits(exponent)
      while iteration >= 0:
         result *= result
         if (exponent >> iteration) & 1:
            result *= base
         iteration -= 1
   else:
         firstModulus = base % modulo
         iteration = bits(exponent)
         while iteration >= 0:
            result = (result * result) % modulo
            if (exponent >> iteration) & 1:
               result = (result * firstModulus) % modulo
            iteration -= 1
   return result
```

この方法では、初期化後は変数 result（とループ変数の iteration）だけが変化していき、他の変数は変化しないという利点がある。このことからハードウェアによる暗号処理の実装を高速かつ安価に実装できる。

base % modulo}} という行は、ちょっとした最適化であり、前の手法にも適用可能である。

これらのバイナリ法では、指数の2進数表現の各ビットごとに自乗の計算が必要であり、そのビットが1であるときだけ乗算も行う。

右向きバイナリ法では、乗算回数をさらに減らす bit windowing optimization や sliding window optimization がある。

また、剰余を求める計算（%）の高速化として、[モンゴメリ乗算](https://ja.wikipedia.org/wiki/モンゴメリ乗算 "wikilink")がある。

## 公開鍵暗号における応用

上述のように、 を法とした  の -冪剰余を求めるには、高々  回の加算、乗算、剰余算が必要である。 加算、乗算、剰余算はそれぞれ被演算数の桁数の多項式時間で計算できる。特に、上述のように乗算を行うたびに剰余演算を行えば、被演算数を常に  未満に保つことができる。よって、 を法とした  の -冪剰余は、入力サイズ  の多項式時間で計算できる。

一方、 から  *b*<sup>*e*</sup> mod *m*}} なる  を求める問題は[離散対数問題](https://ja.wikipedia.org/wiki/離散対数問題 "wikilink")といわれ、効率的な、つまり入力サイズの多項式時間のアルゴリズムは発見されていない。[公開鍵暗号](../Page/公開鍵暗号.md "wikilink")のうちある種のものは、この一方向性を利用して設計されている。
