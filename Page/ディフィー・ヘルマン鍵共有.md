**ディフィー・ヘルマン鍵共有**（ディフィー・ヘルマンかぎきょうゆう、、**DH**）、あるいは**ディフィー・ヘルマン鍵交換**（かぎこうかん）とは、事前の秘密の共有無しに、[盗聴](../Page/盗聴.md "wikilink")の可能性のある通信路を使って、[暗号鍵の共有を可能にする暗号プロトコルである](../Page/鍵_\(暗号\).md "wikilink")。この鍵は、[共通鍵暗号方式](https://ja.wikipedia.org/wiki/共通鍵暗号方式 "wikilink")の鍵として使用可能である。

## 概要

[1976年](../Page/1976年.md "wikilink")にスタンフォード大学の2名の研究員[ホイットフィールド・ディフィー](../Page/ホイットフィールド・ディフィー.md "wikilink")と[マーティン・ヘルマン](https://ja.wikipedia.org/wiki/マーティン・ヘルマン "wikilink")は、[公開鍵暗号](../Page/公開鍵暗号.md "wikilink")の概念を提案し、その具体的な方式の一つとして、ディフィー・ヘルマン鍵共有（DH鍵共有）プロトコルを提案した。この鍵共有方式は共通鍵暗号方式における鍵の受け渡しを安全に行うために提案された方式である。

このプロトコルは、通信を行いたい2者が各々公開鍵と秘密鍵（私有鍵ともいう）を用意し、公開鍵のみを相手に送信し、各自、自分の秘密鍵と受信した公開鍵から共通鍵を作成できる方法である。第三者が送受信されるデータ（すなわち、二人の公開鍵）を盗聴しても鍵を生成することができない所に特徴がある。

[アメリカ合衆国](https://ja.wikipedia.org/wiki/アメリカ合衆国 "wikilink")と[カナダ](https://ja.wikipedia.org/wiki/カナダ "wikilink")で[特許](../Page/特許.md "wikilink")が取得された。[1997年](https://ja.wikipedia.org/wiki/1997年 "wikilink")[4月29日](../Page/4月29日.md "wikilink")に両国でアルゴリズムの特許期限が切れたため、現在では誰でも自由に利用できる。

## プロトコルの内容

この方式は以下のように行われる。まず値の大きな[素数](../Page/素数.md "wikilink") *p* を用意する。また *g* を \(({\mathbb Z}/p{\mathbb Z})^{\ast}\) の生成[元とする](../Page/集合.md "wikilink")。この *g* と *p* は公開されているものとする\[1\]。

いま 太郎くん と 花子さん が通信を行うとする。このとき 太郎くん と 花子さん はお互い秘密の値 *a*, *b* を選択する、この値は 0 以上 *p-2* 以下の中から[ランダム](../Page/ランダム.md "wikilink")に選ぶ。（ここで、ゼロや小さな値（g<sup>a</sup> \< p となる *a* 等）を選択すると安全性が損なわれるが、そのような確率は無視できるほど小さい。）

太郎くん は次の値 *A* を計算してこれを 花子さん に送信する。

\[A = g^a \mod p\]

花子さん も同様に *B* を計算してこれを 太郎くん に送信する。

\[B = g^b \mod p\]

太郎くん は自身の秘密の値 *a* と受信した *B* から以下の値を計算する。

\[K_A = B^a \mod p\]

花子さん も自身の秘密の値 *b* と受信した *A* から以下の値を計算する。

\[K_B = A^b \mod p\]

このとき 太郎くん と 花子さん が計算した K は共に

\[K_A = K_B = g^{ab} \mod p\] になっているため、以後この値を共通鍵暗号方式の鍵として使用する。

ここで第三者 Z がこの二人の通信を盗聴して *A* と *B* を入手しても、\(A = g^a \mod p\) と \(B = g^b \mod p\) から \(K = g^{ab} \mod p\) を[多項式時間](https://ja.wikipedia.org/wiki/多項式時間 "wikilink")で計算する方法は現在の所、存在しないので Z は鍵 K を生成することが困難である。このため 太郎くん と 花子さん が安全に通信を行うことが可能になる。

## 中間者攻撃

ディフィー・ヘルマン鍵共有自体は[認証](../Page/認証.md "wikilink")手段を提供するものではないため、単独では[中間者攻撃](../Page/中間者攻撃.md "wikilink")に対して脆弱である。

ここではディフィー・ヘルマン鍵共有における中間者攻撃の具体的な手順について示す。

[DNS偽装](https://ja.wikipedia.org/wiki/DNS偽装 "wikilink")・[ARPスプーフィング](../Page/ARPスプーフィング.md "wikilink")・その他の手段により攻撃者 Z がこの二人の通信を中継して *A* と *B* を入手したとする。

このとき Z はそれらに対して秘密の値 *c* と *d* を選択する。この値は *a* や *b* と同じ基準で選択される。

Z は *c* を用いて次の値を計算して Y に送信する。

\[A_z = g^c \mod p\]

また *d* を用いて計算して X に送信する。

\[B_z = g^d \mod p\]

Y は自身の秘密の値 *b* と受信した値から以下の値を計算する。

\[K_{BZ} = A_z^b \mod p\]

X は自身の秘密の値 *a* と受信した値から以下の値を計算する。

\[K_{AZ} = B_z^a \mod p\]

Z は自身の秘密の値 *c* と *d* と受信した *A* と *B* から以下の値を計算する。

\[K_{ZAZ} = A^d \mod p\]

\[K_{ZBZ} = B^c \mod p\]

このとき X と Y と Z が計算した K は共に

\[K_{AZ} = K_{ZAZ} = g^{ad} \mod p\]

\[K_{BZ} = K_{ZBZ} = g^{bc} \mod p\] になっている。

以降の通信で Z はこれらの値を共通鍵暗号方式の鍵として使用して X と Y の通信を中継し、盗聴や改ざんを行う。

## 公開鍵の選択

公開鍵は（何かしらの証明を付けた）静的なものであっても、一時的なもの（ephemeral、この場合特に**DHE**と略記される）であってもかまわない。一時的な鍵を使用した場合、鍵そのものには認証がないため、別な方法で認証を行うこととなる。もし認証がなければ、上述の通り[中間者攻撃](../Page/中間者攻撃.md "wikilink")に対して脆弱となる。どちらか一方の鍵が静的なものであった場合、中間者攻撃を受けることはなくなるが、[forward secrecy](https://ja.wikipedia.org/wiki/forward_secrecy "wikilink") のような、その他の高度なセキュリティに与ることはできなくなる。静的な鍵を持つ側では、自身の秘密鍵漏洩を防ぐため、相手の公開鍵を確認して、安全な共通鍵生成関数を利用する必要がある。

共有した秘密をそのまま鍵として使うこともできなくはないが、ディフィー・ヘルマン鍵共有で生成したことによってできる弱いビットの影響を除去するため、秘密をハッシュに通すことが推奨される\[2\]。

## 問題点

### 処理負荷

ディフィー・ヘルマン鍵共有は負荷のかかる処理であり、[SSL/TLSに適用した場合では](../Page/Transport_Layer_Security.md "wikilink")、通常の[RSA暗号](../Page/RSA暗号.md "wikilink")による鍵交換の場合と比較して、サーバの[スループット](../Page/スループット.md "wikilink")が6分の1程度まで落ち込むという実験結果も存在する\[3\]。

### パラメータの設定ミス

これはディフィー・ヘルマン鍵共有のシステム自体に存在する問題ではないが、2013年の調査では、SSL/TLSでディフィー・ヘルマン鍵共有を有効としているサーバのうち、[電子署名](../Page/電子署名.md "wikilink")のビット数よりDHのビット数のほうが小さく、[総当たり攻撃](../Page/総当たり攻撃.md "wikilink")に対して弱くなってしまっているサーバが、実に80%以上の割合で存在していた\[4\]。

### 弱鍵の存在と Logjam 攻撃

原理上は解読がきわめて困難ではあるが、実装上の問題が存在する場合解読が可能になる場合がある。

また原理上、使われている[素数](../Page/素数.md "wikilink")に対して十分な量の事前計算を行えば、その素数に対しては比較的短い時間で鍵を解読することができる (言い換えれば、その素数を弱鍵にすることができる)\[5\]。[2015年](../Page/2015年.md "wikilink")、このことを元にした論文が発表された\[6\]。

この論文において、[Alexaによるトップ](../Page/アレクサ・インターネット.md "wikilink")100万[HTTPS](../Page/HTTPS.md "wikilink")ドメインの中で512ビット輸出版DHEを許可している 8.4% のうち 82% が、1024ビット非輸出版DHEでもトップドメイン全体の 17.9% がそれぞれ単一の素数を使い回しており、これらの素数に対して事前計算 (ある種の線形代数計算を含む) を行うことで多くのサーバーの通信に対して解読が行えることを指摘した。特に512ビットの素数に対しては解読を実証し、数千コアと約8日の事前計算により、およそ70秒で解読できることを示した\[7\]。

加えてサーバーが用いる素数についての[離散対数](../Page/離散対数.md "wikilink")問題を*リアルタイムで*解ける攻撃者が存在した場合、たとえクライアント側が弱いDHEを許可していなくても偽のサーバーに接続させる[中間者攻撃](../Page/中間者攻撃.md "wikilink")が成立し、例えばサーバー側が512ビットの輸出版DHEを許している場合、輸出版DHEを許可していない新しいクライアントであっても通信をダウングレードさせることが可能であることを示した (**Logjam** 攻撃)\[8\]。

同論文は1024ビットの非輸出版DHEについても、数億[ドルのコストをかけ専用ハードウェアを構築した場合](../Page/アメリカ合衆国ドル.md "wikilink")、1年で1つの素数に対して十分な線形代数計算を実行できる可能性があることを示唆している\[9\]。
