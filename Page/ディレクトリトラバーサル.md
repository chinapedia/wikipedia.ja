**ディレクトリトラバーサル** () とは、利用者が供給した入力ファイル名のセキュリティ検証/無害化が不十分であるため、ファイル[APIに対して](https://ja.wikipedia.org/wiki/Application_Programming_Interface "wikilink")「親ディレクトリへの横断 (traverse)」を示すような文字がすり抜けて渡されてしまうような攻撃手法のことである。

この攻撃の目標は、アクセス可能にすることを意図していない[ファイルへのアクセスをアプリケーションに命令することである](../Page/ファイル_\(コンピュータ\).md "wikilink")。この攻撃は、コードに含まれるバグの攻撃とは対照的に、セキュリティの欠如 (ソフトウェアがまさにそう振る舞うことになっている動作) を攻撃する。

**ディレクトリトラバーサル**には `../` (ドットドットスラッシュ) 攻撃\[1\]、 [ディレクトリ](../Page/ディレクトリ.md "wikilink")クライミング、およびバックトラッキングのような別名がある。この攻撃の一部の形態は、[正規化](../Page/正規化.md "wikilink")攻撃でもある。

## 例

[PHPによる脆弱なアプリケーションの典型例を以下に示す](../Page/PHP_\(プログラミング言語\).md "wikilink"):

``` php
<?php
$template = 'blue.php';
if ( is_set( $_COOKIE['TEMPLATE'] ) )
   $template = $_COOKIE['TEMPLATE'];
include ( "/home/users/phpguru/templates/" . $template );
?>
```

このシステムに対する攻撃として、以下のようなHTTP要求の送信が考えられる:

    GET /vulnerable.php HTTP/1.0
    Cookie: TEMPLATE=../../../../../../../../../etc/passwd

生成されるサーバの応答は以下のようになる:

``` http
HTTP/1.0 200 OK
Content-Type: text/html
Server: Apache

root:fi3sED95ibqR6:0:1:System Operator:/:/bin/ksh
daemon:*:1:1::/tmp:
phpguru:f8fk3j1OIf31.:182:100:Developer:/home/users/phpguru/:/bin/csh
```

`/home/users/phpguru/templates/` の後に繰り返される `../` 文字列が[ルートディレクトリ](../Page/ルートディレクトリ.md "wikilink")へさかのぼる[include()](https://www.php.net/manual/en/function.include.php)を引き起こし、それからUNIXパスワードファイル [/etc/passwd](https://ja.wikipedia.org/wiki/passwd "wikilink") をインクルードする。

UNIX [/etc/passwd](https://ja.wikipedia.org/wiki/passwd "wikilink") は[クラッカーがパスワード](../Page/クラッカー_\(コンピュータセキュリティ\).md "wikilink")[クラッキングにしばしば使うため](../Page/クラッキング_\(コンピューター用語\).md "wikilink")、**ディレクトリトラバーサル**の実演でよく使われるファイルである。

## ディレクトリトラバーサルの変化形

ディレクトリトラバーサルを防ぐのは見かけより困難である。「既知のまずい文字を取り除く」防御戦略は**失敗する**可能性が高い。

ディレクトリトラバーサルが実際に機能するかどうか決定する他の入り組んだ要因が多数存在する。しかしながら、もしアプリケーションがパラメータなどの正当性を検証しなかったら、攻撃者がこの機能性を悪用する小さな隙間を見つけ出す可能性は非常に高い。

以下は既知のディレクトリトラバーサル攻撃文字列の一部である:

### UNIXにおけるディレクトリトラバーサル

[UNIX](../Page/UNIX.md "wikilink")ライクOSに共通のディレクトリトラバーサルは `../` 文字列を使う。

### Microsoft Windowsにおけるディレクトリトラバーサル

[Microsoft Windowsや](https://ja.wikipedia.org/wiki/Microsoft_Windows "wikilink")[DOSのディレクトリトラバーサルは](../Page/DOS_\(OS\).md "wikilink") `..\` 文字列を使う。

は、多くのWindowsプログラムやAPIが[UNIX](../Page/UNIX.md "wikilink")ライクなディレクトリトラバーサル文字列も受け付ける。

各[パーティション](../Page/パーティション.md "wikilink")（[ドライブ](../Page/ドライブレター.md "wikilink")）は独立したルートディレクトリ (Cドライブの場合 `C:\` とラベル付けされる) を持ち、その上に共通のルートディレクトリは存在しない。このため、Windows上のディレクトリトラバーサル脆弱性のほとんどは攻撃先が1パーティションに限られることになる。

### パーセントエンコードされたディレクトリトラバーサル

正規化問題。

クエリ文字列は通常使用する前にURIデコードされるが、行儀の悪いWebアプリケーションの中には、ディレクトリトラバーサルを防ぐためURIデコード前のクエリ文字列から以下のような危険な文字列を走査するものが存在する:

  - `..`
  - `..\`
  - `../`

しかし、このような「デコード前に走査する」アプリケーションは以下のような[パーセントエンコードされたディレクトリトラバーサルに対して脆弱である](../Page/パーセントエンコーディング.md "wikilink"):

  - `%2e%2e%2f` (`../` に変換される)
  - `%2e%2e/` (`../` に変換される)
  - `..%2f` (`../` に変換される)
  - `%2e%2e%5c` (`..\` に変換される)

etc.

### Unicode / UTF-8エンコードされたディレクトリトラバーサル

正規化問題。

[UTF-8](../Page/UTF-8.md "wikilink")は[ブルース・シュナイアー](https://ja.wikipedia.org/wiki/ブルース・シュナイアー "wikilink") (Bruce Schneier) とジェフリー・シュトレイフリング (Jeffrey Streifling) による[Cryptogram Newsletter July 2000](https://www.schneier.com/crypto-gram/archives/2000/0715.html)において脆弱性と攻撃媒介の源として注目された。

Microsoftが自社のWebサーバに[Unicode](../Page/Unicode.md "wikilink")対応を追加したとき、`../` をエンコードするための新しい方法が彼らのソースコードに入り込み、ディレクトリトラバーサル防止の試みが迂回されてしまう原因となった。その結果、

  - `%c1%1c`
  - `%c0%9v`
  - `%c0%af`

のような複数のパーセントエンコードが `/` や `\` の文字に変換された。

これはなぜかというと、パーセントエンコードはMicrosoftのWebサーバにより、対応する8ビット文字にデコードされたからである。Windowsと[DOSは](../Page/DOS_\(OS\).md "wikilink")[ASCII](../Page/ASCII.md "wikilink")を基にした[正規化](../Page/正規化.md "wikilink")された8ビット文字集合を伝統的に使っていたため、これは歴史的には正しい動作だった。

しかしながら、初期の[UTF-8](../Page/UTF-8.md "wikilink")は正規化されていなかったものの、今や数種類の文字列が、同じ文字列に変換されうる文字列エンコーディングになった。MicrosoftはUTF-8を[正規化](../Page/正規化.md "wikilink")することなくトラバーサル対策チェックを行っていたため、([HEX](https://ja.wikipedia.org/wiki/16進数 "wikilink")) `C0AF` と ([HEX](https://ja.wikipedia.org/wiki/16進数 "wikilink")) `2F` が[文字列](../Page/文字列.md "wikilink")比較を行うとき同じ文字となることに気付かなかった。

## ディレクトリトラバーサルを防ぎうる手法

ディレクトリトラバーサルを防げる可能性のあるアルゴリズムは以下のようなものである:

  - URIの要求がファイルの要求にならないように処理する。たとえば、後続の処理を続ける前にユーザーコード内にフックを実行する。
  - ファイルやディレクトリへのURI要求がなされるべきときは、存在するならファイルやディレクトリへのフルパスを構築し、すべての文字を正規化する (たとえば、`%20` をスペースに変換する)。
  - 「文書ルート」は完全修飾されており、正規化されており、パスは既知で、この長さNの文字列であると仮定する。このディレクトリ外のファイルは提供できないと想定する。
  - 要求されたファイルへの完全修飾パスの先頭N文字が「文書ルート」と正確に同じであることを確認する。
  - もしそうなら、ファイルを返すことを許す。
  - そうでなければ、要求は明らかにWebサーバが提供することを許されているもの領域を踏み越えているので、エラーを返す。

## 関連項目

  - [Chroot jailsは](../Page/Chroot.md "wikilink")、作成方法が正しくないとディレクトリトラバーサルの影響を受けるかもしれない。ディレクトリトラバーサルの攻撃媒介として、jail外のディレクトリ上にあるファイルを開いた[ファイル記述子](../Page/ファイル記述子.md "wikilink")がありうる。作業ディレクトリはもう一つの攻撃媒介になりうる。
